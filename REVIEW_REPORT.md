当前版本已经具备完整个人财务产品形态：短信入库、流水查询、消费分析、预算、归属和标签、自动规则、目标、CSV 导出、密码 Session 登录，以及桌面端和移动端界面。

基础工程状态正常：

- `go test -race ./...` 通过。
- `npm run build` 通过。
- Review 时 Git 工作区干净。
- 已使用隔离数据库完成关键接口复现，没有操作正式数据。

当前没有编译或启动级阻断，但仍有 3 项会直接影响财务数据或核心交互的 P1 问题。建议完成 P1 后再定义为最终稳定版；P2 可分批处理。

## 2. 优先级定义

| 级别 | 含义 |
|---|---|
| P0 | 服务不可用、数据大规模丢失或直接安全失守，必须立即处理 |
| P1 | 会产生错误财务数据、写错用户配置，或破坏核心操作流程，发布前应处理 |

| P3 | 维护性、体验一致性或技术债，可后续优化 |

## 3. P1：发布前应处理

### P1-1 相同短信重复上报会重复计账

涉及位置：

- `internal/service/service.go:45-112`
- `internal/store/store.go:51-101`

当前行为：

每次 `POST /api/v1/sms` 都直接创建一条 `raw_sms`，解析成功后再创建一条 `transactions`。数据库中没有短信指纹、上报请求 ID 或业务唯一约束。

隔离环境复现：

1. 连续提交两次内容完全相同的 ¥100.50 消费短信。
2. 服务分别返回交易 ID 1 和 2。
3. 月消费统计变成 ¥201.00，`total_txn_count=2`。

影响：

- iPhone 快捷指令重试、用户重复触发、网络超时后的补发都会重复记账。
- 所有预算、趋势、日报、余额安全垫和人员统计都会被污染。
- 目前前端没有删除或合并交易功能，产生重复数据后难以自行修复。

建议方案：

- 首选：由客户端发送稳定的 `idempotency_key`，服务端建立唯一索引。
- 兼容现有快捷指令：服务端计算规范化短信指纹，例如 `SHA-256(source + normalized_text)`，并配置合理的去重时间窗。
- 如果业务允许银行产生正文完全相同但确属不同交易的短信，指纹应组合解析后的卡尾号、金额、方向、发生时间、商户和原文摘要，避免永久按全文误去重。
- 原始短信落库和交易创建应处于同一事务，或者明确设计可恢复状态，避免交易已创建但状态更新失败后客户端重试造成重复。

验收标准：

- 同一个幂等键连续或并发提交两次，只产生一条交易。
- 第二次请求返回可识别的 `duplicate` 状态或原交易信息，而不是新增交易。
- 不同交易即使金额和商户相同，也不会被误判为重复。
- 增加 Service/API/Store 层并发去重测试。

### P1-2 点击刷新会把 React 点击事件作为搜索关键词

涉及位置：

- `web/src/App.jsx:177`
- `web/src/App.jsx:301-332`
- `web/src/App.jsx:837-865`

当前行为：

首页、流水页和分析页使用 `onClick={onRefresh}`。React 会把点击事件作为第一个参数传入，但 `loadAll` 的第一个参数定义为 `search`：

```jsx
const loadAll = useCallback(async (search = data.search) => {
  // ...
  api.transactions({ q: search, limit: 80 })
})
```

结果是刷新请求可能携带：

```text
q=[object Object]
```

影响：

- 首页最近流水会突然变空。
- 流水页搜索状态会被污染成 `[object Object]`。
- 用户会认为刷新导致数据消失。

建议方案：

- 所有 React 点击入口包装为 `onClick={() => onRefresh()}`；或者
- 将 `loadAll` 改为不接收搜索参数，搜索使用独立函数；或者
- 对参数做严格类型检查，只接受字符串。

验收标准：

- 首页、流水页和分析页点击刷新后，交易请求不包含异常 `q`。
- 已有搜索词在刷新后的保留/清除行为有明确一致的产品定义。
- 添加前端测试，模拟按钮点击并断言 API 参数。

### P1-3 查看历史月份时，预算可能写入错误月份

涉及位置：

- `web/src/App.jsx:629`
- `web/src/App.jsx:841-865`

当前行为：

- 页面数据根据 `period.month` 加载预算。
- 设置页保存预算时却固定发送 `month: currentMonth()`。

典型场景：

1. 用户在首页或分析页切换到历史月份。
2. 进入设置页，页面显示历史月份预算。
3. 点击保存，实际写入当前月份。
4. 刷新仍读取历史月份，因此刚保存的预算看不到。

影响：

- 用户会认为预算无法保存。
- 实际数据被写入另一个月份，可能覆盖本月预算。

建议方案：

- 设置页明确展示并选择预算月份。
- 保存和读取必须使用同一个月份状态。
- 如果产品规定预算只能设置本月，则进入设置页时应固定展示本月预算并明确说明，不应沿用分析页月份。

验收标准：

- 页面显示的预算月份、请求中的月份和刷新后的列表月份完全一致。
- 从历史月份进入设置页不会静默写入当前月份。
- 增加跨月份保存测试。

## 4. P2：建议近期处理

### P2-1 允许选择未来月份，但后端会产生反向时间区间

涉及位置：

- `web/src/App.jsx:164-172`
- `web/src/App.jsx:337-346`
- `internal/store/analytics.go:285-304`

当前行为：

前端允许不断点击“下个月”或直接选择未来月份。后端将结束日期截断到今天，但没有同步处理仍在未来的开始日期。

隔离环境实测，在 2026-07-18 查询 2026-08 返回：

```text
from=2026-08-01
to=2026-07-18
days=1
```

建议方案：

- 产品层禁止选择未来月份，并禁用“下个月”按钮；同时
- 后端作为防线，在截断后重新校验 `from <= to`，否则返回明确的 400，或统一返回合法空区间。

验收标准：

- API 永远不返回 `from > to`。
- UI 无法进入无意义的未来月份，或能明确展示“暂无数据”而不伪造区间。

### P2-2 `SESSION_SECRET` 配置没有实际作用

涉及位置：

- `internal/config/config.go:25-26, 48, 65-72`
- `cmd/server/main.go:44`
- `deploy/env.example:14-15`

当前行为：

配置层读取或生成 `SESSION_SECRET`，部署文档称其用于 Cookie signing，但创建 Session Store 时没有传入该值。Session 实际只是进程内存中的随机 ID 映射。

影响：

- 无论是否配置 `SESSION_SECRET`，服务重启都会让所有 Session 失效。
- 页面“会记住会话”的承诺只在同一个进程生命周期内成立。
- 配置和部署文档会误导运维人员。

建议方案二选一：

1. 实现由 `SESSION_SECRET` 签名的无状态 Cookie，并正确处理撤销；或
2. 将 Session 存入 SQLite，Cookie 仅保存随机 Session ID；或
3. 如果接受重启后重新登录，则删除无效的 `SESSION_SECRET` 配置和 signing 描述，明确说明 Session 为进程内状态。

验收标准：

- 配置、实际行为和部署文档一致。
- 如果宣称 Session 可持续 30 天，正常服务重启后仍应保持登录。

### P2-3 CORS 反射任意 Origin 并允许凭证

涉及位置：

- `internal/api/middleware.go:34-51`

当前行为：

服务端将任意请求 `Origin` 原样写入 `Access-Control-Allow-Origin`，同时设置：

```text
Access-Control-Allow-Credentials: true
```

生产环境由同一 Go 服务提供前端和 API，本身不需要开放任意跨域。若部署环境存在同站子域、代理或其他浏览器会携带 Cookie 的场景，恶意 Origin 可能读取或修改管理数据。

建议方案：

- 生产环境默认不发送 CORS 响应头。
- 开发环境只允许明确地址，例如 `http://localhost:5173` 和实际局域网开发地址。
- 加入 `Vary: Origin`。
- 不允许将请求 Origin 无条件反射。

验收标准：

- 未在白名单中的 Origin 无法读取带凭证 API。
- 同源生产页面正常工作。
- Vite 开发代理正常工作。

### P2-4 后端支持 Token-only 管理，Web UI 却无法登录

涉及位置：

- `internal/config/config.go:62-64`
- `web/src/App.jsx:68-78`
- `README.md` 的鉴权说明

当前行为：

后端允许只配置 `ADMIN_TOKEN` 或兼容的 `API_TOKEN`。但没有 `ADMIN_PASSWORD` 时，React 登录页只显示“无法登录”，已删除 Token 输入入口。

建议方案二选一：

1. 产品明确规定 Web 必须使用密码登录：启动时要求 `ADMIN_PASSWORD`，Token 仅供脚本使用，并同步修改 README；或
2. 恢复 Web Token 登录，但应避免长期保存在 `localStorage`，并说明安全权衡。

验收标准：

- 所有允许启动的鉴权配置都有明确可用的管理路径。
- README、部署说明、后端校验和登录页面说法一致。

### P2-5 “全部流水”实际上最多只显示 80 笔

涉及位置：

- `web/src/api/client.js:56-62`
- `web/src/App.jsx:844`
- `web/src/App.jsx:284-318`

当前行为：

前端固定请求 `limit=80`，没有分页、加载更多或总数提示，但页面标题是“全部流水”。历史数据超过 80 笔后，用户无法从 UI 查看更早记录。

建议方案：

- 增加基于 `offset` 的分页或无限滚动。
- API 返回 `total`、`limit` 和 `offset`。
- 搜索结果同样支持继续加载。

验收标准：

- 用户可以从 UI 访问所有流水，而不只是最新 80 条。
- 页面显示总数和当前加载范围。

### P2-6 CSV 导出的时区与应用配置不一致

涉及位置：

- `internal/api/features.go:15-24`
- `internal/api/features.go:50`

当前行为：

应用使用 `TZ_NAME` 进行短信解析和统计，但 CSV 使用系统的 `time.Local` 格式化发生时间；默认导出日期也使用系统时区。systemd 示例只配置 `TZ_NAME`，不会自动修改 Go 的 `time.Local`。

影响：

- 服务器系统为 UTC、应用配置为 `Asia/Shanghai` 时，CSV 时间可能少 8 小时。
- 北京时间午夜附近的默认导出日期可能偏一天。

建议方案：

- CSV Handler 使用 Service 配置的 Location，或由 Service 直接返回格式化所需时区信息。
- 默认日期范围也基于同一个 Location。

验收标准：

- UI、统计区间和 CSV 中同一笔交易显示相同的本地日期时间。
- 增加 UTC 服务器 + Asia/Shanghai 应用时区测试。

### P2-7 首页的零值回退会显示错误口径

涉及位置：

- `web/src/App.jsx:222-225`

当前代码使用逻辑或：

```jsx
d.today_consume || a.today?.expense
d.unlabeled_week || a.unlabeled_count || 0
```

当真实值为 `0` 时会进入后面的回退值：

- 今日消费为 0、但存在转账支出时，“今天花了”可能显示转账金额。
- 近 7 日待整理为 0 时，首页可能回退显示全历史未整理数量。

建议使用 `??`，或者明确分别展示当前周与全部历史口径。

验收标准：

- 合法零值必须显示为 0，不触发其他统计口径。
- 首页数字说明和后端字段含义一致。

### P2-8 异步错误处理不完整，失败可能被显示成成功空状态

涉及位置：

- `web/src/App.jsx:491-493`
- `web/src/App.jsx:659-745`
- `web/src/App.jsx:758-770`

当前行为：

- 整理页加载失败被空 `catch` 吞掉，`queue` 保持空数组，页面可能显示“已经整理完了”。
- 创建目标、成员、标签、规则、删除操作和 CSV 导出多数没有统一 `try/catch`。
- 失败后可能出现未处理 Promise，用户没有明确反馈。

建议方案：

- 建立统一 mutation helper，处理 busy、成功提示、错误提示和 401。
- 空数据、加载失败和真正完成应使用不同 UI 状态。
- 删除成员、标签、规则和目标增加确认，尤其删除成员会清空已有流水归属。

验收标准：

- 任意接口返回 4xx、5xx 或断网时，页面有明确错误信息。
- 加载失败不会显示“已经整理完了”。
- mutation 进行中不可重复点击。

### P2-9 多银行卡情况下，“账户余额”和目标进度语义不成立

涉及位置：

- `internal/store/store.go:309-318`
- `internal/store/features.go:393-440`

当前行为：

`LatestBalance` 从所有交易中取最近一条带余额的记录，没有按银行卡分组，也没有求和。首页将该值称为“账户余额”，所有存钱目标也直接使用这一笔余额。

如果系统只跟踪一张主卡，当前实现可以接受；如果会接收多张卡，则该数字只是“最近收到短信的那张卡余额”，不是总资产。

需要产品方先确认语义：

- 单主卡模式：界面明确显示银行和卡尾号。
- 多卡模式：分别展示各卡最新余额，再计算可选的总余额；需注意信用卡余额与储蓄卡余额不能直接同义相加。
- 存钱目标最好维护独立进度或绑定具体账户，而不是所有目标都等于同一个最新余额。

验收标准：

- 多卡数据下不会把单卡余额误称为总账户余额。
- 目标进度来源对用户可解释。

## 5. P3：维护性与产品完整性

### P3-1 未解析短信在 React UI 中没有入口

后端保留 `/api/v1/unparsed`，前端 Client 也定义了 `api.unparsed()`，但没有任何页面调用。README 写明“解析失败可查看”，当前只能直接调 API。

建议在设置或整理页增加“解析失败”入口，至少展示原文、失败原因、来源和时间。

### P3-2 存在已废弃 Token 状态和导入

`App.jsx` 仍导入 `setToken`，设置页仍创建 `tokenInput` state，但页面已删除对应控件。建议清理，避免后续维护者误判 Token 登录仍受支持。

### P3-3 API 输入校验不够严格

建议补充：

- 预算月份必须符合 `YYYY-MM` 或明确支持的 `*`。
- 预算 `kind` 只能取允许枚举。
- 分析 `kind` 只能取允许枚举。
- 流水的非法 `from/to` 当前会被静默忽略，应返回 400。
- 规则 `match_field` 和 `match_op` 应验证枚举，不应将未知字段默认为商户、未知操作默认为等于。

### P3-4 财务金额长期建议使用整数分

当前金额使用 SQLite `REAL` 和 Go `float64`，展示前会 round2。对目前个人规模通常可用，但长期累计、导入和比例计算仍可能出现浮点误差。后续若加入手工编辑、账单核对或多币种，建议迁移为整数最小货币单位，或使用十进制定点实现。

## 6. 已接受风险：登录全局锁定

产品方明确接受当前策略：连续错误密码达到阈值后，全局阻止新的密码登录 30 分钟。

代码确认：

- 全局锁只在 `POST /api/v1/auth/login` 的 LoginGuard 中生效。
- 已登录 Session 的管理请求走 `Sessions.Valid`，不会检查 LoginGuard。
- 触发全局登录锁不会清空现有 Session，也不会影响已经登录的用户继续查看和操作。

因此本项不列为待修复问题。

保留说明：

- 锁定期间，Session 已过期、清除 Cookie、更换设备或尚未登录的用户无法进入。
- 服务重启会重置 LoginGuard，但当前也会同时丢失所有内存 Session。
- 如果未来用户量或公网暴露范围扩大，应重新评估此策略。

## 9. 最终验收建议

- P1 全部关闭。
- API 不会因重试制造重复交易。
- 所有月份相关读写使用同一明确口径。
